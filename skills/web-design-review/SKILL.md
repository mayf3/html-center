# 网页设计评审

## 总览

用户要求评审浏览器产品时使用这个 skill，例如落地页、SaaS、管理后台、wiki、仪表盘、移动 Web、文档站。

输出应该是评审，不是泛泛总结。要具体、有严重程度、有证据、有判断。中文用户默认输出中文；长评审、需要分享、需要沉淀或用户要求 HTML 时，优先生成一个可直接打开的单文件 HTML 评审报告。

## 决策原则

规则只负责筛选和组织，不能替代判断。

最终决策交给 LLM：

- 综合证据
- 权衡 tradeoff
- 判断什么最重要
- 直接给出结论

## HTML 产物原则

参考单文件 HTML 报告的做法：不要把长评审压成一墙 markdown；把证据、问题、风险、修复顺序做成可浏览、可跳转、可分享的页面。

默认要求：

- 生成自包含 `.html` 文件：内联 CSS/少量 JS，无构建步骤，直接用浏览器打开。
- 页面语言默认 `zh-CN`，导航、标题、标签、空态、按钮、表格列名都尽量中文。
- 桌面端必须使用左侧导航：`aside`/`nav` 固定在主内容左侧，包含锚点和状态计数；不要用顶部 pill 导航替代。
- 移动端允许左侧导航折叠为顶部目录或横向滚动，但内容顺序仍按左侧导航的章节组织。
- 章节必须有稳定锚点，导航项至少包含：总诊断、严重问题、主要问题、细节问题、修复优先级、证据、实现风险。
- 用卡片、表格、标签、截图、简图或代码片段呈现空间信息；避免只有连续段落。
- 保持工作型界面：信息密度高、层级清楚、颜色克制；不要做营销页、装饰性 hero、渐变球或无意义插画。
- 如果包含代码、DOM、文件位置或截图标注，把证据放在问题旁边，而不是只放在附录。

## HTML Center 发布

生成 HTML 报告后，默认尝试发布到常驻 HTML Center，避免每份报告单独启动一个 Web 服务。

- 服务地址优先读取 `HTML_CENTER_URL`，未设置时使用 `http://127.0.0.1:4177`。
- 单文件报告上传该 `.html` 文件；模块化评审包上传整个目录，并用 `--entry` 指定入口，例如 `index.html` 或 `docs/index.html`。
- 类别默认使用 `design-review`；可按内容改为 `ux-review`、`architecture-review`、`implementation-review` 等稳定分类。
- 上传成功后，最终回复同时给出本地文件路径和 HTML Center URL；上传失败不阻断交付，但要说明失败原因。

发布脚本：

```bash
python <skill-dir>/scripts/publish_html_report.py ./design-review.html \
  --title "Design review" \
  --category design-review \
  --source "<review target>"
```

模块化目录示例：

```bash
python <skill-dir>/scripts/publish_html_report.py ./docs \
  --title "System design review" \
  --category architecture-review \
  --entry index.html
```

## 模块化系统评审

当评审对象不是单个页面，而是产品系统、代码库、架构设计、项目参考对比或“准备自己实现一个类似产品”时，不要只输出一篇总评。默认生成模块化评审包：

- 一个全局索引：`docs/index.html` 或用户指定的 HTML 入口，列出所有模块、诊断、优先级和跳转链接。
- 一个总览文档：说明产品目标、参考项目、核心取舍、推荐路线。
- 多个模块文档：至少覆盖接入层、gateway、agent runtime、workflow/core、tools/审批/安全、记忆/session/search、插件/skills、配置/状态存储、UI/控制台、测试/发布/运维。
- 每个模块都要包含：参考项目观察、可借鉴设计、不建议照搬点、MyClaw 推荐设计、MVP 边界、后续阶段、关键风险、验收标准。
- 对模块之间的依赖关系要显式写出，避免各页只是独立观点。
- 如果用户要求“分阶段实现”，每个模块必须标注进入 Phase 0/1/2/... 的最小实现切片。

模块化评审包的 HTML 索引不替代模块文档；它负责导航、摘要和优先级，让用户能从全局跳到细节页。

推荐 HTML 骨架：

```html
<html lang="zh-CN">
  <body>
    <div class="review-shell">
      <aside class="side-nav">
        <nav aria-label="评审导航">
          <a href="#summary">总诊断</a>
          <a href="#critical">严重问题</a>
          <a href="#major">主要问题</a>
          <a href="#minor">细节问题</a>
          <a href="#priority">修复优先级</a>
          <a href="#evidence">证据</a>
          <a href="#risk">实现风险</a>
        </nav>
      </aside>
      <main>
        <section id="summary">...</section>
      </main>
    </div>
  </body>
</html>
```

## 评审流程

1. 识别产品目标、主要用户、核心任务。
2. 先看渲染结果：截图、浏览器页面、响应式断点、真实数据状态。
3. 再看信息架构：导航、页面层级、内容分组、任务流。
4. 再看视觉系统：排版、对比度、间距、密度、一致性。
5. 检查交互质量：可点击暗示、状态、空态、错误、加载、响应式、键盘可访问性。
6. 检查实现风险：内容规模、边界情况、数据加载、性能、可维护性。
7. 最后给出严苛但可执行的结论：什么坏了，什么只是弱，下一步该做什么。

## 输出要求

- 先给一句诊断，直接说明当前体验最大的问题。
- 按严重程度列问题：严重、主要、细节；必要时可在括号里保留 critical、major、minor。
- 能定位就给具体位置：文件、组件、屏幕区域、DOM landmark。
- 区分产品问题和实现问题。
- 最后给优先修复项，以及最小但收益最高的下一步。
- 如果生成 HTML 报告，文件名默认使用 `design-review.html` 或用户指定路径；最终回复给出文件路径和验证方式。
- 如果 HTML Center 可用，生成 HTML 报告后发布到中心；最终回复给出中心入口 URL。

## 语言

默认使用用户语言。如果用户用中文，或产品服务中文工作流，界面标签、导航、空态、错误、评审输出都优先中文。只有在产品面向英文用户、源码术语无法准确翻译或用户明确要求英文时，才保留英文。

## 评审标准

默认从这些角度看：

- 清晰度：新用户 10 秒内能否说出这是什么。
- 流程：最短路径是否明显、低摩擦。
- 密度：页面是否太空、太挤，或状态之间密度不一致。
- 层级：视线是否落在该落的位置。
- 一致性：组件、间距、标签是否像一个系统。
- 信任：来源、新鲜度、状态是否清楚。
- 可访问性：对比度、焦点顺序、语义、减弱动画、可读字号。
- 鲁棒性：空态、长标题、大数据集、坏链接、离线状态。

## HTML 报告检查

交付 HTML 报告前检查：

- 左侧导航在桌面端确实位于左侧，并能跳转到各章节。
- 中文标签覆盖导航、章节、严重程度、表格、按钮和空态。
- 每个严重/主要问题都有影响、证据和修复建议。
- 页面可在无构建步骤下打开；样式、布局、锚点和响应式状态正常。
- 长文本、长文件名、窄屏不会互相遮挡。
- 颜色不依赖单一色相，不用装饰性背景干扰阅读。

## 评审设计文档时

如果评审对象是系统设计页、架构页或 design review 工作台，必须检查是否有这些图：

- 数据流图：输入、处理、存储、输出。
- 代码依赖图：前端、后端、数据目录、外部服务之间的依赖。
- 状态流图：关键 pipeline 的状态转换。
- 请求-响应图：从用户动作、浏览器、服务端、数据层到返回结果的完整路径。
- 风险图：当前阻塞、主要风险、下一步修复优先级。

还要检查是否展示项目目录结构、关键代码职责、代码行数，以及评审结果是否能保存为历史记录。缺少这些内容时，把它列为主要问题；缺少持久化评审历史时，如果产品目标是持续改进，把它列为严重或主要问题，由 LLM 根据影响判断。

## 评审代码时

如果有源码，先读源码再判断 UI：

- `index.html`、路由、页面组件、布局壳层、主题 token。
- 页面状态模型、加载路径、数据契约。
- 搜索、过滤、导航逻辑。

如果有截图或运行中的 app，先看渲染结果，再回到代码验证。

## 验证

- 对本地 Web app，启动或复用开发服务器，用浏览器检查桌面和移动断点。
- 对静态 HTML 评审报告，至少用浏览器或截图工具确认非空、导航可见、内容未重叠。
- 如果无法运行或截图，明确说明原因，并把评审限定在可见证据内。

## 语气

直接、具体、公平。避免空泛表扬。有效就说明为什么有效，失败就说明哪里失败、怎么失败。
