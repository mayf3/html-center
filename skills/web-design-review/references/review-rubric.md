# 评审 Rubric

## 输入检查表

- 产品目标
- 主要用户
- 目标设备和断点
- 截图、录屏或可运行 URL
- 相关代码路径
- 用户语言和产品 locale
- 是否需要可分享的 HTML 评审报告

## 严重程度

- 严重（Critical）：阻塞理解、任务完成、信任或核心转化。
- 主要（Major）：明显损害可用性、层级、效率、可访问性或可维护性。
- 细节（Minor）：不会阻塞主流程，但影响专业度、一致性或长期质量。

## 默认评审顺序

1. 产品清晰度
2. 信息架构
3. 视觉层级
4. 交互状态
5. 可访问性
6. 性能和鲁棒性
7. 可维护性
8. 当产品是设计评审或治理工具时，检查评审结果是否能持久化

## 判断策略

- Rubric 负责筛选和组织，不替代判断。
- 最终结论由 LLM 综合证据后给出。
- 如果两条规则冲突，优先服务用户任务和产品目标。
- 默认让 UI 语言匹配用户工作流。
- 对设计文档，默认要求数据流、依赖、状态、请求-响应、风险、目录和职责视图；除非这些视图与范围明显无关。

## HTML 报告验收

当输出 HTML 评审报告时，默认验收这些点：

- 单文件、可直接打开、无构建步骤。
- 桌面端左侧导航，移动端可折叠或横向滚动。
- 中文优先：导航、章节、标签、表格列名、空态和修复建议都尽量中文。
- 章节锚点完整：总诊断、严重问题、主要问题、细节问题、修复优先级、证据、实现风险。
- 问题卡片必须包含：问题、影响、证据、建议、位置。
- 风险表必须区分产品风险、交互风险、实现风险。
- 长文本、长文件名、窄屏、空数据和大量问题都不破坏布局。

```text
严重问题
- 问题、影响、证据、修复建议

主要问题
- 问题、影响、证据、修复建议

细节问题
- 问题、影响、证据、修复建议

1. ...
2. ...
3. ...
```
